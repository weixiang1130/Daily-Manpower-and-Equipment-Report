# 節點 03：查核獨立性與下拉選單化

**日期**：2026-07-02
**狀態**：已被節點 04 進一步調整（回報區塊延遲顯示的精神延續，UI 實作方式已變）

## 背景（使用者原始需求）

1. 申請跟回報查核理論上應該要有所區分：填單當下如果有人貪圖方便一口氣把申請跟回報都填完，會讓查核失去意義
2. 任何填寫的部分都應該是下拉式選單，不應該用「點選」的方式讓使用者填寫；下拉選單若沒有需要的選項，最下面要能新增新選項
3. 「目前工地」下拉旁要能選擇「今天是哪一位工程師填報」
4. 送出以後如果要編輯或刪除，清單要有明確的編輯、刪除按鈕，點編輯後要把完整資料帶回上方填寫區塊

## 設計決策

### 查核獨立性

在節點 02，「出工回報」區塊雖然存在，但新增申請時就已經看得到、填得到。改為：**新增申請時完全隱藏回報區塊**，欄位預設值（`actual=0`、`reportDone` 未勾）必然導致新建立的紀錄狀態是「待回報」。回報只能透過「點選既有紀錄進入編輯」才能填寫——結構上排除了「一次送出就把兩階段做完」的可能性，而不只是道德勸說。

另外加入軟性提醒：若「稽核人員」與「申請人」為同一人，送出時跳出警告 toast（不強制擋下，避免小團隊人力調度上的死結）。

### 下拉選單化

原本工作地點/工作內容類別/機具類型是 checkbox 群組（`.chip-group`），改為原生 `<select multiple size="6">`。分包商、申請人、稽核人員、負責工程師等也統一使用 `<select>`，並在選單最後加入「＋ 新增選項…」，選到這個選項時跳出 `prompt()` 輸入新值，寫回 `STATE.config`，並即時套用到全站所有相同欄位（單一資料來源）。

### 目前填報工程師

頁首新增「目前填報工程師」下拉，選定後自動帶入新申請的「申請人」欄位，減少重複選擇。

## 變更內容

- `index.html`：`l_reportSection` / `e_reportSection` 包一層 div，預設 `display:none`，編輯既有紀錄時才顯示
- `app.js`：新增 `handleAddNewOption()`、`ADD_NEW` sentinel、頁首 `currentEngineer` 邏輯
- 清單每列改為明確的「編輯」「刪除」按鈕，取代整列可點擊的隱性互動

## 已知限制（後續節點改善）

- 回報欄位仍是自訂設計（稽核人員、三道查核、簽單狀態），尚未逐欄對照既有 Excel 查核表的實際欄位
- 申請與回報雖然視覺上有隱藏機制，但實際上仍是**同一張表單**，不是兩個真正獨立的模組
- `<select multiple>` 雖然技術上是下拉選單，但視覺呈現是多列清單框，與分包商/申請人的單行下拉觀感不一致
