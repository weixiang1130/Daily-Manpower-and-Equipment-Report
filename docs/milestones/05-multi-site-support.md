# 節點 05：多工地列控支援

**日期**：2026-07-02
**狀態**：現行架構（本文件撰寫當下的最新版本）

## 背景（使用者原始需求）

工地的部分，希望能根據不同工地分門別類進行列控——不一定要放在頁首右上角的全域選單，而是**申請的時候才選擇是為哪一個工地填寫**。總覽也應該要根據不同工地別，分別顯示目前機具跟點工出工的狀態與情況。使用者並提供了公司目前需要列控的 16 個實際工地名稱。

> 為避免客戶專案代號外流，實際工地清單未收錄於本文件與程式碼庫，僅記錄於使用者本機環境；程式碼中的 `DEFAULT_CONFIG.sites` 改為通用範例值，實際名單請於系統「設定」頁面輸入。

## 設計決策

### 工地從「全域狀態」改為「逐筆欄位」

節點 01–04 的 `STATE.currentSite` 是一個全域切換器，所有清單、表單、總覽都跟著這一個值走。這與使用者的實際工作模式不符：同一位成本控制人員一天內可能要為好幾個不同工地填報，「先選好一個工地再開始填」的模式反而是多餘的一層操作。

因此把「工地」變成申請表單裡的一個必填欄位（跟分包商、申請人同一層級），資料模型上 `labor[]` / `equipment[]` 每筆紀錄本來就有 `site` 屬性（節點 01 就有），只是現在填寫時機從「頁首預先選定」改成「申請當下選擇」。

### 總覽改為跨工地彙總 + 逐工地列控表

- 「整體概況」卡片改為彙總**全部工地**的數字（不再是單一工地）
- 新增「各工地列控總覽」表格，把設定裡的每個工地都列一列（即使目前沒有任何紀錄也列出來，讓使用者一眼看出哪些工地完全沒有回報，這本身也是一種異常訊號），欄位包含：點工待回報、機具待回報、本月出工回報次數、本月人數異常、簽單尚未繳回
- 點擊工地名稱可直接跳轉到「點工申請/回報」頁籤，並自動把清單篩選器設為該工地——把「總覽發現問題」到「進去看細節」的路徑縮短成一次點擊

### 清單與報表新增工地篩選器

點工清單、機具清單、歷程報表分別維護各自的篩選狀態（`laborListFilter` / `equipListFilter` / `reportSiteFilter`，記憶體變數、不寫入 `localStorage`，每次重新整理頁面預設回到「全部工地」），預設顯示全部工地資料，可切換至單一工地檢視。CSV 匯出檔名與內容會依目前選擇的篩選工地調整。

## 變更內容

- `index.html`：移除頁首 `<div class="site-picker">`；`點工申請`、`機具申請`表單各自新增「工地」`<select>`；點工清單、機具清單、歷程報表新增篩選工地下拉
- `app.js`：
  - `DEFAULT_CONFIG.sites` 改為通用範例值（實際 16 個工地名單僅存於使用者本機 `localStorage`，經由設定頁輸入）
  - `STORAGE_KEY` 由 `v4` 升為 `v5`，移除 `STATE.currentSite`
  - 新增 `renderSiteBreakdown()`、`jumpToSite()`、`ALL_SITES` sentinel
  - `renderLaborList()` / `renderEquipList()` / `REPORT_DEFS` 改用各自的篩選變數而非全域 `currentSite`
- 工地欄位比照其他名稱類欄位，同樣支援「＋ 新增選項」（新工地開案時可直接在表單裡新增，不必先進設定頁）

## 已知限制（後續可考慮的方向）

- 目前資料仍僅存在單一瀏覽器的 `localStorage`，多工地情境下若有多人同時在不同裝置填報，彼此看不到對方的資料——這是下一步若要多人協作，需要接後端資料庫（例如 Firebase/Supabase 或公司內部 API）的主要動機
- 各工地列控表目前是固定欄位，尚未支援依欄位排序或篩選異常件數 > 0 的工地
