# 節點 10：共用資料庫與填錯工地防呆（v8）

**日期**：2026-07-03
**狀態**：現行架構（本文件撰寫當下的最新版本）

## 背景（使用者原始需求）

1. 各自地端填寫的內容其他人看不到——每個人開起來畫面都不一樣。理論上應該是共編：大家填完，其他人都能看到填寫內容與總覽，而不是只存在自己裝置上
2. 因為任何人都可以切換編輯其他工地，需要防呆機制確保填寫者沒有填錯工地，避免數據大亂

## 1. 共用資料庫（Netlify Functions + Blobs）

業務資料從 localStorage 全面改存雲端共用資料庫，所有使用者讀寫同一份資料。選擇 Netlify Blobs 的原因：**不需要新帳號、不需要新服務**——就在既有的 kgmanpower Netlify 專案內，免費方案內建，也沿用既有的整站 Basic Auth 保護。

### 儲存結構（Blobs store: `audit-data`）

| Key | 內容 |
|---|---|
| `master` | 全域工地清單 `{sites:[...]}` |
| `cfg:<encodeURI(工地)>` | 該工地基礎資料（分包商/人員/地點/類別/機具/點工人員池） |
| `rec:<encodeURI(工地)>:<labor\|equipment>:<id>` | 單筆紀錄（父單含子層 report） |

**逐筆紀錄各自一把 key**：不同紀錄同時寫入互不影響；同一筆被兩人同時編輯時為後寫者覆蓋（last-write-wins），以現場人數與填報頻率屬可接受風險。

### API（`netlify/functions/api.mjs`，路徑 `/api/data`）

- `GET ?scope=all`：master + 全部工地的 config 與紀錄（開站/重新整理用）
- `GET ?site=X`：單一工地資料
- `POST {op: master|config|record|deleteRecord|clearSite|clearAll, ...}`：各類寫入
- 驗證：與整站 Edge Function 相同的 Basic Auth（函式內再驗一次作第二道防線，防止繞過邊緣層直打函式路徑）
- Blobs 以 `consistency: "strong"` 開啟，確保讀到最新寫入

### 前端資料流

- 開站 `boot()`：GET scope=all → 填入記憶體快取 → 首次使用時自動種入預設名單（master 不存在→寫入；工地 config 不存在→以 config.local.js 名單種入）
- 寫入採 **await-first**：先確認雲端寫入成功，才更新畫面並清空表單；失敗時表單內容保留並提示重試，避免「以為送出了其實沒有」
- 讀取更新：頁首「↻ 重新整理」按鈕＋切換到總覽/點工/機具頁籤時靜默重抓（表單編輯中則跳過，避免蓋掉輸入中的內容）
- localStorage 已完全退場（僅 sessionStorage 記住本分頁的工地選擇與管理員狀態）

## 2. 填錯工地防呆（三道機制）

1. **開站攔截頁**：每次開啟頁面（新分頁/新工作階段）必須先在整頁攔截畫面明確點選本次要填報的工地，不給任何預設值——「明確選擇」取代「沿用上次」
2. **常駐工地徽章**：四個表單面板（點工申請/回報、機具申請/回報）標題旁常駐琥珀色「📍 目前工地」徽章，填寫過程中隨時可見
3. **送出前確認**：點工/機具「申請」送出時彈出確認視窗，明列即將寫入的工地名稱與單據摘要，按下確認才寫入（回報是從清單點選既有單據進入、風險低，不另加確認）

## 已知限制

- 同筆紀錄並發編輯為 last-write-wins，無合併/鎖定機制
- 無使用者身分紀錄（誰改的不留痕跡）；若日後需要稽核軌跡（audit log），建議升級至 Supabase 等含驗證的後端
- 本機以靜態伺服器開啟時無 API，會顯示「無法連線」畫面——本機開發需用 `netlify dev` 或直接以部署網址測試
