# 節點 14：安全補強——CSV 公式注入與 record.id 跳脫（v11.1）

**日期**：2026-07-07
**狀態**：現行架構

## 背景

v11 上線後進行安全審查，兩項純程式可修的弱點立即處理（第三項「伺服器端破壞性操作無權限分級」需搭配環境變數 ADMIN_TOKEN，另案待裁示）：

1. **CSV 公式注入**：匯出 CSV 未中和以 `=` `+` `-` `@` 開頭的儲存格；廠商名、備註、查核回饋皆為自由輸入，惡意內容在 Excel 開啟時會被當公式執行
2. **record.id 未跳脫**：清單按鈕 `data-id` 直接插入 id；API 端接受任意 id 字串，持帳密者可寫入惡意 id 對其他使用者造成儲存型 XSS（例如藉此釣 adminPin），且含 `:`／`%` 的 id 會破壞 Blobs key 結構

## 變更內容

- `app.js` `downloadCSV()`：非純數字卻以 `=` `+` `-` `@` 開頭的儲存格前置 `'`（純數字如差異欄 `-1` 不受影響，Excel 仍視為數字）
- `app.js` 點工/機具清單：六處 `data-id` 改經 `esc()`（dataset 讀取時實體自動解碼，按鈕行為不變）
- `api.mjs`：`record` 與 `deleteRecord` 兩個 op 伺服器端驗證 id 格式 `^[A-Za-z0-9_-]{1,64}$`（`uid()` 產生的 id 為純 base36，既有資料全數相容）

## 驗證紀錄

本機（狀態注入）：
- 注入惡意 id `x" onmouseover="alert(1)`：HTML 中正確跳脫為 `&quot;`、DOM 無被注入的事件屬性、`dataset.id` 往返讀取正常（功能不受影響）
- CSV 匯出攔截 blob 檢查：`=1+1公司` → `'=1+1公司`、`@SUM(A1:A9)`／`+886…`／`-負開頭文字` 皆前置 `'`；差異欄 `-1` 維持純數字
