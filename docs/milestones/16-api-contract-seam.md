# 節點 16：API 合約正式化與後端替換餘裕（IT 標準對接準備）

**日期**：2026-07-07
**狀態**：現行架構

## 背景

IT 部門評估地端部署後回覆：現行「檔案/Blobs 直存」的資料層不符公司標準，未來需依公司標準重寫「存資料到後端」這段。使用者指示：**先留好餘裕**，讓未來能以最小成本按 IT 標準改寫。

分析：前端本來就只透過單一函式 `api()` 呼叫單一端點 `/api/data`——接縫天然存在，但**只存在於程式碼裡，沒有正式規格**。重寫後端的人必須逆向工程兩份實作才能知道合約。留餘裕的正解不是預先改寫資料層（IT 標準未知，猜了會白做），而是**把接縫正式化成規格文件**，讓任何語言/資料庫的後端都能照規格實作、以現行前端直接驗收。

## 變更內容

1. **`docs/API-CONTRACT.md`（核心交付）**：前後端接縫合約 v1——傳輸與驗證、GET 兩種讀取、POST 七個操作的完整 request/response 與驗證規則、409 樂觀並發語意（明定不可更改）、全部資料模型欄位字典（含 v10/v11 欄位演進與舊欄位相容規則）、給後端重寫者的六條相容須知（整筆覆寫語意、未知欄位保留、時區、參考資料表設計、驗收方式）
2. **`app.js` API 位址可配置**：新增 `API_BASE`，可由 `config.local.js` 的 `apiBase` 覆寫（預設 `/api/data` 不變）——公司後端掛任何路徑都不用改程式；`api()` 函式加上「唯一接縫」的說明註解指向合約文件
3. **`DEPLOYMENT.md` 新增 §7「未來改接公司標準後端」**：告訴 IT 重寫時只需實作合約、如何驗收；原測試聲明改為 §8
4. `CLAUDE.md` 檔案地圖標記合約文件為「改欄位先改這份」的變更紀律

## 設計決策 / 取捨

- **不預先重寫資料層**：IT 標準（語言/資料庫/中介層）尚未給出，此時改寫是瞎猜；合約文件＋可配置接縫是「不做白工的最大餘裕」
- **合約以 Markdown 而非 OpenAPI**：單端點+op 分派的 RPC 形態用 OpenAPI 表達彆扭且多一份要同步的檔案；文件中欄位字典已足供 IT 產 schema，若 IT 要求 OpenAPI 可依此轉出
- **變更紀律入文件**：之後每次改版動到欄位，合約文件必須同步（已寫進合約開頭與 CLAUDE.md）

## 影響範圍

- `app.js`：僅 API 層頂部（新增常數與註解，行為不變——預設路徑相同）
- 新增 `docs/API-CONTRACT.md`；`DEPLOYMENT.md`、`CLAUDE.md` 補述
- 雲端/地端兩份後端實作皆不動

## 驗證紀錄

- 本機預覽（靜態伺服器）：重載後無 console 錯誤；以 fetch 攔截確認 `api()` 目標為 `/api/data?...`（預設路徑與 query 組合皆與改版前一致，零迴歸）；覆寫邏輯為單行表達式 `(LOCAL.apiBase && String(LOCAL.apiBase)) || "/api/data"`，以程式碼審視確認
- 合約內容與兩份參考實作（api.mjs / server.mjs）逐案核對（op/狀態碼/驗證規則/欄位）
