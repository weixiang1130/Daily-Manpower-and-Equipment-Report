## 交付前審查與硬化（同分支第二 commit）

交付前對本包執行多代理程式審查（10 個搜索視角＋分組驗證，多數發現以 LocalDB 實測證實），共確認 13 項問題並全數修復：

- **匯入安全**：產出腳本加 `SET XACT_ABORT ON`（實測原本約束錯誤只跳單句、其餘照常 COMMIT 且 exit 0——「部分匯入卻回報成功」）；文件旗標統一為 `-f 65001 -x -b -C`（-x 防 `$(環境變數)` 被靜默注入自由文字、-b 遇錯即停、-C 自簽憑證）
- **孤兒工地**：sites 改從 master∪stores 聯集建檔，已移除工地標 `is_active=0`、歷史紀錄保留（原版會 NULL site_id 靜默漏失）
- **髒資料防護**：壞 id（合約 regex）／壞日期（空字串原會靜默存成 1900-01-01）／壞狀態／缺工種名 → 跳過＋分類計數回報；大小寫/空白重複選項去重（CI 定序撞 UNIQUE）
- **資料保真**：空陣列保留 `[]` 不塌 NULL（合約 string[] 型別）；輸出不再翻譯字串內嵌換行；updatedAt 由 UTC 轉本地（避免同欄混存兩種基準）；備註/內容欄放寬 NVARCHAR(MAX)（前端無字數上限）
- **Schema**：`total_ot` 改計算欄位（杜絕三欄漂移，需 QUOTED_IDENTIFIER ON——DDL 與產出腳本已內建）；id/contracted 加 CHECK；補 IX_equip_vendor；彙總 VIEW 補「工作內容/機具類型」彙集欄（OPENJSON＋STRING_AGG，對齊 UI 計價彙總）
- **文件**：合約補「舊單 totalOT 歸前 2 小時段」正準規則、is_active 與合約行為的關係、白名單正準宣告；DEPLOYMENT §7 連結本包；切換指令補建庫步驟並以 DEPLOYMENT §4 為完整流程

**第二輪驗證**：合成備份（含機具已回報/待回報、孤兒工地、壞 id/日期/狀態、缺工種名、大小寫重複選項、600 字備註、多行回饋、`$(PATH)` 文字、空陣列、UTC 時間戳）→ 轉換器跳過 6 筆並正確分類計數，其餘匯入 exit 0；庫內十項行為逐一驗證通過（含機具三表與兩機具 VIEW 首次行使）。正式備份迴歸：筆數與彙總與第一輪完全一致，無跳過資料。

## 設計決策 / 取捨

- **API 薄層不代寫**：開發機無 .NET SDK 無從實測，且資訊處的框架/認證（AD?）標準未知——交「已實測的資料層＋合約＋對照表」比交「沒跑過的程式」對雙方都負責
- **多值欄位 JSON 過渡**：遷移成本最低且合約允許；正規化留給資訊處依報表需求決定
