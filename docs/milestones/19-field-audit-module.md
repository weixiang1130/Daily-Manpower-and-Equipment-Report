# 節點 19 — 成控現場稽核模組（v13）

## 背景

成本管理部有定期赴工地抽查點租工的稽核作業（比對申請/簽單與現場點名、人臉辨識、門禁卡等紀錄），過去以人工紀錄＋簡報彙整。使用者要求在系統內新增「成控現場稽核」模組：

1. 需管理員密碼進入，且**工地人員完全看不到此畫面**（稽核方＝成控；被稽核方＝工地）
2. 連動申請父層：選日期／廠商 → 帶出該申請單（工作內容/地點/申請數）逐項查核
3. 查核採**二元判定**：每項必選「相符／不相符」，不相符自動展開必填的原因欄
4. 不限單據狀態（待回報也可稽核），**以申請數為基準**現場清點實際人數/台數
5. 稽核結果可**匯出 PDF** 供製作抽查報告

## 變更內容

- **新頁籤「成控現場稽核」**（`index.html`）：非管理員完全隱藏（`hidden`），登出管理員時若停留在此頁自動跳回總覽；頁籤點擊另有 isAdmin 防禦
- **稽核流程**（`app.js` 新區塊）：工地/類型（點工/機具）/日期/廠商 四層連動 → 申請單卡片列表 → 稽核表單：
  - 申請數（唯讀快照）＋現場實點數 → 差異自動計算
  - 逐項查核：相符/不相符切換；不相符展開原因欄（必填）；全部項目必選才可儲存
  - 現場狀況說明（不限字數）＋稽核人（sessionStorage 記憶上次輸入）
- **查核項目可配置**：預設清單點工 7 項（點名人數/人臉辨識/白卡/工項/地點/同廠商重複計價/簽單核對）、機具 5 項；可由 `config.local.js` 的 `auditItems: {labor:[], equipment:[]}` 覆蓋，未來增減項目免改版
- **資料模型**：稽核存於單據 `audits[]` 陣列（一單可多次稽核），沿用 `op:record` 整筆覆寫＋baseV 版本檢查——**雲端後端零修改、零部署**（僅前端＋靜態檔）。合約新增 §4.5
- **稽核紀錄清單**：本工地點工＋機具合併、稽核期間篩選、**單筆編輯**（原內容載回表單、更新取代——保留原稽核日期與申請數快照、另記 `editedAt` 編輯日並於清單/報告標示）、單筆刪除（confirm）
- **匯出**：PDF（列印視圖＋瀏覽器另存 PDF，零套件；含彙總表＋逐筆明細，仿既有抽查簡報結構；封面資訊與彙總表均列**稽核人員**、結尾附稽核/覆核**簽章欄**）＋ CSV（含編輯日期欄）
- **SQL 交付包同步**：`DB-SCHEMA.sql` 新增 `labor_audits`/`equip_audits` 兩表（items 以 JSON 過渡，ISJSON CHECK）＋ `v_audit_log` 合併檢視（含 OPENJSON 算不符項數）；`backup-json-to-sql.py` 新增 audits 匯出（含 id/日期驗證與跳過計數）；README 對照表更新（10 表 5 VIEW）

## 設計決策 / 取捨

1. **audits 內嵌於單據 vs 獨立命名空間**：選內嵌——不動 API 介面（後端零修改，部署凍結下重要）、與版本衝突保護天然相容；代價是資料層面工地使用者技術上可經 API 讀到稽核內容（UI 完全隱藏），真正資料權限隔離留待地端 AD 分權
2. **PDF 用列印視圖而非 PDF 函式庫**：維持零依賴原則；瀏覽器「另存為 PDF」跨裝置可用
3. **二元判定（無「未查」態）**：依使用者裁示；未選視為未完成、擋儲存，確保每份稽核逐項有結論
4. **稽核不限單據狀態**：現場稽核可能先於回報發生；`statusAtAudit` 快照留證
5. **applied 存快照**：申請數在稽核後仍可能被改，稽核當下的基準必須固定

## 驗證紀錄

- 本機預覽（靜態伺服器＋fetch stub 注入示範資料，未碰共用資料庫）：
  - 點工流程：差異自動計算（8 申請/6 實點→−2）、未填稽核人擋下、未選項目擋下、不相符展開原因欄且未填擋下、儲存成功（v 版本遞增、audits[]=1、紀錄清單即時出現）
  - 機具流程：標籤正確切為「申請台數/實點台數」、5 項查核、儲存成功
  - PDF HTML：含彙總表、不符原因、動態字串全部 esc() 跳脫；刪除稽核（confirm）後清單即時更新
  - 權限：登出管理員→頁籤隱藏＋自動跳回總覽；console 零錯誤
- SQL 層（LocalDB 重建 KG_AUDIT_TEST）：新 DDL 一次跑通（15 物件＝10 表＋5 VIEW）；稽核 fixture 匯入——好資料 2 筆入庫、壞 id/壞日期 2 筆跳過並計數；`v_audit_log` 不符項數（OPENJSON）計算正確；多行說明與單引號原樣保真

## 第二輪硬化（2026-07-21，上線後 MAX 十角度審查 12 項確認發現全數修復）

- **最重要**：pickAuditRecord/editAudit 的 refetchSite 原本無條件整批刷新快取，會讓其他頁籤編輯中表單送出時 baseV 漂移、繞過 409——改以 `otherFormEditing()` 把關（其他表單編輯中時跳過刷新；稽核儲存自身仍受 baseV/409 保護）
